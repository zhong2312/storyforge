> 目的:让仓库接手者(尤其 Codex,看不到 WPS)拿到软硬结合一致性工程化的**问题 / 现状 / 思路 / 方案 / 可行性**全貌。
> 关系:本文件是 ROADMAP 一致性轨道(`CONSISTENCY-*`)与 `AI-COPILOT-DESIGN.md` §2.2 检测环的**总纲**;`TRANSPARENT-GENERATION-PIPELINE.md` 的节点 gate 复用这里定义的校验器。

## 0. 北极星:一件事,三头受益

把"一致性"从提示词概率链,做成**确定性工程**。这一件事同时让三件事受益:

1. **bug 变少** —— 很多反复出现的 bug,根因是一致性没兜住(状态/物品/认知/时序对不上)。
2. **长期一致性变好** —— 写到几百章不崩(人物动机、设定、伏笔、时间线自洽)。
3. **游戏基座地基** —— "作者级自洽 ⟷ 运行时 per-player 涌现的无限自由"这个游戏化命题,前提就是先有确定性的一致性地基(见 VISION v3)。

> 一句话:不头疼医头。做一致性工程化,bug、长期一致性、游戏基座三头一起受益。

---

## 1. 问题 / 现状(为什么必须做)

**现状**:一致性目前**全靠一条概率链**——
```
抽取(软) → 拼进 prompt(软) → 劝模型(软) → LLM 事后审校(软) → 给作者肉眼看(人)
```
**除已落地的 `held-items`(物品重复获得)这一块外,几乎没有一环用代码"算"出"违反了已确立的事实 / 规则"。**

- 抽取:AI 可能漏、可能错。
- 拼 prompt:不分主次一股脑打包,重点淹没、token 失控。
- 劝模型:提示词写了"别剧透",模型可能不听。
- LLM 审校:会漏硬矛盾、烧 token、"劝不判"(自我说服"看着还行")。
- 作者肉眼:最后也是唯一的确定性闸门,但累、易漏。

**根本局限**:确定性判决目前只有 `held-items` 一块,**主干远未成型**,绝大多数一致性仍散在提示词(而非代码判决)——这就是"自己说修了实际无效"类坑的温床。

## 2. 核心思路:软硬分界 + 作者确认闸门

把每个 AI 动作拆成三种角色(三色):

| 角色 | 干什么 | 例子 |
|---|---|---|
| **软 · AI 造零件(不确定)** | 生成、抽取、找矛盾建议、语义召回 | 写正文/续写、抽事实/状态/物品、摘要/交接、一致性审校建议、embedding 检索 |
| **硬 · 代码车床+质检(确定)** | 算、簿记、溯源、隔离、失效传播 | 章序解析、未来不泄漏过滤、世界隔离、当前有效事实投影、持有物投影、受控谓词守卫、写回校验、引文逐字回查、hash/CAS 溯源、上下文预算、影响传播 |
| **闸门 · 作者确认(软→硬)** | 软零件经确认变 canon 硬事实 | 采纳 adopt |

**铁律原则**(与 `AI-COPILOT-DESIGN.md` §2.2 一致):
> "**有没有违反已确立的事实 / 规则**"必须由**确定性代码**判(零 token、不漏硬矛盾);**向量化 / LLM 只负责召回相关材料和提出建议**,不作为判定一致性的依据。

理由:让 LLM "看"一致性,会放大旧问题(劝不判、漏硬矛盾、烧 token)。确定性检测既准又省 token,是多模型 agent 成本可控的关键。

## 3. 收敛路线(第 0 ~ 5 步 · 严格串行)

### 第 0 步 · 止血(bug-batch)
把已知一致性类 bug 先修掉,别让它继续坑用户。对应 ROADMAP 各社区反馈批次。**进行中。**

### 第 1 步 · 物品 / 状态账本硬校验 ✅ 已完成(CONSISTENCY-1)
`held-items`:从 `itemLedger` 投影"当前持有物",硬核对"已持有物品又被写成首次获得"。**第一块确定性校验器,已落地并合入 main。** 验证了"事件流水 → 投影 → 确定性硬校验"这套模式在本项目跑得通。

### 第 2 步 · 上下文分主次
生成前上下文不再一股脑打包;按需 / 分层 / 预算裁剪,重点信息突出。对应 `assembleContext` 现有分层 + `TRANSPARENT-GENERATION-PIPELINE`(把拼接透明化、可编辑)。

### 第 3 步 · 设定 ↔ 正文强绑定 + 修改联动
正文 `hash / CAS` 溯源;设定/正文改了 → 派生记忆 `stale` + 影响分析标记后续章待复核。`consistency/impact-analysis.ts` 已有雏形,补全为强绑定。

### 第 4 步 · canon 校验器完全体 + 世界宪法 + 时轴抽象
把确定性校验器补齐成一套:
- **认知 / 知识账本(CONSISTENCY-2)** —— 每角色知识状态,硬检测"开天眼"(见 ROADMAP CONSISTENCY-2)。
- 受控谓词守卫(`fact-predicate-registry`)、当前有效事实投影(`readCurrentFacts`)、世界规则 canon validator、历史锚点 validator、引文逐字回查。
- **世界宪法**:不可违反的硬规则集合。
- **时轴抽象**:章序 / 时间线的规范表示(`resolveCanonicalChapterSequence` 是起点)。

### 第 5 步 · 游戏基座自然掉出
上面都硬化后,"作者级自洽 ⟷ 运行时 per-player 生成"的游戏基座命题,是这套地基的自然产物,不需要单独另起炉灶(见 VISION v3)。

---

## 4. 可行性

- **已验证**:第 1 步(CONSISTENCY-1)落地 + 362 测试绿,证明"事件 → 投影 → 硬校验"模式可行。
- **可复制**:第 4 步的认知账本(CONSISTENCY-2)= 第 1 步同模式,已知可做。
- **不另起炉灶**:每一步都在现有**三注册表 + store + adapter** 上加一层确定性校验,`check:architecture` 守着。
- **风险**:确定性校验必须**精准**(误报伤体验)。所以只**硬判"确定能判的"**(章序/持有/认知边界/世界隔离/未来泄漏);模糊的(文采/动机好坏/反套路)留 **advisory**(软建议),不当硬闸门。
